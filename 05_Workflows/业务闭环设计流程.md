# 业务闭环设计流程

这个流程用于回答一个很常见的问题：

- 现有业务需求能不能跑通闭环
- 适不适合交给 agent 或自动化系统推进
- 哪些环节可以自动执行，哪些环节必须人工兜底
- 如何把一次性分析沉淀成别人也能复用的设计文档

它不是只服务某一个具体项目，而是一套可以迁移到不同业务里的通用方法。

## 适用场景

当用户提出下面这些诉求时，默认进入这套流程：

- 想评估现有业务是否可以做成 agent 闭环
- 想把现有系统改造成“目标 -> 执行 -> 反馈 -> 判断 -> 下一步”的循环
- 想梳理业务需求到底卡在输入、动作、反馈还是回退能力
- 想产出一份给团队、合作方或其他 agent 参考的闭环设计文档
- 想把内容业务、运营流程、客服流程、知识库流程等改造成更稳定的工作流

## 核心判断

判断一个业务能不能闭环，先看四件事：

1. 输入是否明确
2. 动作是否可执行
3. 结果是否可观察
4. 失败是否可回退

如果这四项都很弱，先不要谈自动化闭环，应该先补基础能力。

如果至少前三项已经比较清晰，就可以先做“半自动闭环”。

如果四项都已经具备，再考虑做“高自动化闭环”。

## 闭环最小结构

一个可运行的业务闭环，至少应该有下面 7 个部分：

1. 目标
2. 输入
3. 执行动作
4. 结果反馈
5. 判断规则
6. 状态流转
7. 异常处理与人工接管

可以把它理解成这个最小链路：

`目标 -> 输入 -> 动作 -> 结果 -> 判断 -> 状态更新 -> 下一步`

## 标准设计步骤

### 1. 先定义业务目标

先写清楚这次闭环到底要解决什么问题，不要一上来就讨论工具。

至少回答下面三个问题：

- 这条业务链最终想交付什么结果
- 成功的判断标准是什么
- 哪些内容不在这条闭环里处理

如果目标本身说不清，后面设计的状态机和验收规则通常都会失真。

### 2. 盘点现有业务链路

先按真实业务写出现状，不要直接写理想流程。

建议先列出：

- 当前入口有哪些
- 业务现在是怎么被触发的
- 每一步由谁执行
- 每一步依赖哪些系统、脚本、页面或人工判断
- 当前最常见的卡点在哪里

这一步的目的不是“画漂亮流程图”，而是找出真正能落地的环节和真正的阻塞点。

### 3. 拆成输入、动作、反馈三层

这是最重要的一步。

#### 输入层

写清楚业务的输入来自哪里，例如：

- 用户提交的表单
- 一个链接或本地文件
- 一条工单
- 数据报表
- 定时触发任务
- 上一流程产出的结构化结果

输入层需要回答：

- 输入格式是否稳定
- 输入字段是否足够
- 是否需要预处理、清洗、补字段、分类

#### 动作层

写清楚系统到底要“做什么”。

动作通常包括：

- 调用脚本
- 访问 API
- 读写数据库
- 生成文档
- 更新状态
- 调用外部服务
- 触发人工审批

动作层需要回答：

- 哪些动作已经有执行入口
- 哪些动作还停留在人脑或手工操作
- 哪些动作可自动重试
- 哪些动作一旦失败会产生高风险

#### 反馈层

写清楚系统做完以后，怎么知道它做成了没有。

反馈可以是：

- 生成了某个文件
- 某条记录状态更新成功
- 某个页面出现了预期结果
- 指标达到阈值
- 人工审核通过
- 下游系统返回成功回执

没有反馈，就不是闭环，只是“执行了一次动作”。

### 4. 设计判断规则

闭环要想稳定运行，必须把判断写成明确规则，而不是留给模糊直觉。

至少要写清楚：

- 什么叫成功
- 什么叫失败
- 什么叫可重试
- 什么叫需要人工介入
- 什么叫可以继续推进下一步

如果判断规则过于模糊，agent 很容易在错误路径上持续放大错误。

### 5. 设计状态机

建议每条业务闭环都显式定义状态，而不是只写“做完了没”。

一个最小状态机通常包括：

- `new`
- `processing`
- `blocked`
- `waiting_human`
- `done`
- `failed`

如果业务链较长，可以再增加：

- `ready_for_review`
- `approved`
- `rework`
- `published`
- `archived`

状态机的关键不是状态名字，而是写清楚：

- 谁能触发状态变更
- 变更前要满足什么条件
- 失败后回到哪里
- 哪些状态不能自动跨越

### 6. 找出人工闸门

闭环并不等于“完全无人值守”。

很多业务最稳的形态是：

`agent 主导执行 + 关键节点人工确认`

优先把下面这些环节设为人工闸门：

- 高风险事实确认
- 对外正式发布
- 品牌口径定稿
- 付款、发信、投放、删除等不可逆动作
- 涉及账号权限、客户承诺或合规责任的动作

### 7. 明确异常处理策略

没有异常策略的闭环，通常只是在“正常情况里看起来能跑”。

至少要写：

- 常见失败类型
- 每种失败是否允许重试
- 重试次数和间隔
- 超过阈值后谁来接管
- 是否需要保留审计日志

### 8. 分层定义自动化等级

不要默认一步到位做全自动。

建议按下面四档判断：

- `L0 手工流转`：主要靠人执行，系统只做记录
- `L1 辅助闭环`：agent 提供建议和产物，人工推动状态前进
- `L2 半自动闭环`：agent 能自动执行大部分动作，关键节点人工确认
- `L3 高自动化闭环`：大部分流程自动完成，仅异常或高风险节点人工介入

多数团队最适合先落在 `L1` 或 `L2`。

## 一份好用的闭环文档应该包含什么

为了让别人能按自己的业务重用，不要只写结论，建议至少保留下面这些部分：

1. 业务背景
2. 目标与边界
3. 当前现状
4. 目标闭环
5. 输入清单
6. 动作清单
7. 反馈与验收规则
8. 状态机
9. 人工闸门
10. 风险与异常处理
11. 当前可落地部分
12. 还缺的基础设施
13. 建议的下一步迭代

## 推荐输出格式

为了便于跨团队复用，建议把结果写成三层：

### 第一层：一句话判断

例如：

- 当前业务已具备半自动闭环条件，建议先做 `L2`
- 当前业务输入和反馈不稳定，暂不适合直接闭环
- 当前业务可以闭环，但必须保留人工审核节点

### 第二层：结构化设计

至少写清楚：

- 输入
- 动作
- 反馈
- 状态
- 风险

### 第三层：落地清单

例如：

- 先补任务状态表
- 先统一输入模板
- 先把关键动作脚本化
- 先加人工审批节点
- 先补失败重试和日志

## 在内容型业务中的映射

对于当前这个仓库，内容业务闭环通常可以映射成：

- 目标：把来源整理成可复用的主题和可发布内容
- 输入：链接、本地文件、主题要求、已有草稿
- 动作：自动入库、摘要、主题归并、成稿、发布包装、预览检查
- 反馈：生成 Markdown、SVG、PNG、预览页、状态说明
- 判断：是否结构完整、是否具备发布条件、是否需要人工审核
- 状态：`new -> processing -> ready_for_review -> approved -> published`

这类业务通常很适合先做成 `L2 半自动闭环`。

## 什么时候值得继续 skill 化

如果某一类闭环设计任务反复出现，并且满足下面任意两项，就应考虑沉淀为 skill：

- 输入模板稳定
- 输出模板稳定
- 有 3 步以上固定分析流程
- 需要固定参考文档、脚本或模板
- 交给不同 agent 执行也应该得到大致一致的结果

## 最小使用方式

以后当有人问“这个业务能不能闭环”“现有系统能不能跑通”时，默认按下面顺序输出：

1. 先给一句话判断
2. 再给输入、动作、反馈、状态的结构化拆解
3. 再区分已经能跑通的部分和仍然缺失的部分
4. 最后给一个从 `L1` 或 `L2` 开始的落地建议
