# 发布建议——Claude Code 内部 12 组件

## 推荐主标题

**Claude Code 内部 12 组件 — 造车人视角**

> 选这个的理由——主标题点出"内部"和"组件"两个钩子（吸 AI 编程进阶用户），副标"造车人视角"暗示是引擎盖下的视角，避开"深度拆解"这种 AI 标题感。

## 备选标题（按场景排序）

1. **打开 Claude Code 的引擎盖：12 章源码读完，发现一个常见误读**
   - 适用场景——公众号 / 知乎首发，长标题带"打开引擎盖"感官词，吸"想看真东西"那批读者

2. **读完 Claude Code 12 章源码，4000 行里 0 个 if-else**
   - 适用场景——技术社群、X / Twitter 转发，"0 个 if-else"是钩子型反直觉数字，自带传播力

3. **Claude Code 不是 12 个组件，是 2 层架构**
   - 适用场景——朋友圈、小群转发，"不是…是…"句式适合短场景，争议感强

4. **会话短命，磁盘长命：Claude Code 后半段 7 章在干什么**
   - 适用场景——技术博客、长文订阅渠道，金句体标题，吸资深 agent 工程师

5. **抄 Claude Code 时该抄什么：12 章源码归类后的判断**
   - 适用场景——开发者群体，"抄什么"是直接价值钩子，适合自建 agent 那批读者

## 标题选择建议

- **想最大点击率** → 用 2 号（"4000 行里 0 个 if-else"）。"0" 这个数字组合反直觉判断是天然钩子。
- **公众号长文** → 用主标题（造车人视角）。视角型标题在公众号转化率比"干货分享"型高。
- **知乎** → 用 1 号（打开引擎盖）。知乎读者吃"原来是这样"的揭示感，但要避开"揭秘 / 深度拆解"这种烂大街词，"打开引擎盖"更具体。
- **朋友圈/小群转发** → 用 3 号。短、判断明确、有争议感，朋友看到会想点开看你为什么这么说。
- **不要用** "深度拆解 Claude Code 内部架构"这种标题。AI 味浓、"深度"是空范围词、"拆解"是 AI 高频动词，转化率最差。

## 推荐导语

> 你天天用 Claude Code，可能从来没想过它内部到底有几个组件。读完 GitHub 上 shareAI-lab 的 `learn-claude-code` 12 章源码（4000 多行 Python），最大的收获不是"原来这么写"，是发现外界普遍流传的那张"12 组件平铺图"是错的——前 5 章和后 7 章是两个不同物种。

## 推荐分享开场口径（社群转发口径）

> 推荐这篇 - shareAI-lab 在 GitHub 上把 Claude Code 拆成了 12 节课，每章一个机制，加起来 4000 行 Python。我读完最大的发现是这 12 章其实是 2 层（前 5 单 agent 内核、后 7 多 agent 协调底座）。文章里直接引了 s01 循环、s05 SKILL.md 两层加载、s11 身份重注入这几段代码，看完你大概知道自建 agent 时哪几章该抄、哪几章别碰。

## 推荐结尾金句（备选 3 条）

1. **当你的 agent 需要长出新行为时，先问自己——这是该加一个工具，还是该改一层协调骨架。十次有九次，答案是前者。**

2. **工具壳易迭代，架构判断会沉淀。Claude Code 这个产品本身可能两年后就被新东西覆盖，但"循环属于 agent，机制属于 harness"这种判断会留下来。**

3. **会话短命，磁盘长命——这一句概括了 Claude Code 后半段 7 章在解决的全部问题。**

## 不建议使用的发布方式

- **不要发"AI 编程入门"或"Claude Code 上手教程"类标签下**。这篇是给已经用熟的人看引擎盖，错频道。
- **不要切成系列连载**（"Claude Code 内部架构（一）"）。一次性读完比分 5 篇更有效，强行拆会丢主线判断（2 层架构 + Agency 来自训练）。
- **不要在标题上加"独家""首发""万字干货""深度长文"**。AI 味重、跟正文反 AI 味气质相悖。
- **不要单独把代码截图发出来配一句话**。代码片段离开"内层 vs 外层"的对照框架就只是源码引用，没有判断价值。
- **不要让封面图（如果做的话）只放 12 个组件名字铺满**。这正好是文章批判的"12 平铺图"，封面应该用本文配的"2 层架构图"——视觉上立刻传递反误读的判断。
