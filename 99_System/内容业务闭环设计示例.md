# 内容知识库与内容生产闭环设计示例

## 一句话判断

当前仓库已经具备明显的 `L2 半自动闭环` 条件，适合先实现“agent 主导执行 + 关键节点人工确认”。来源入库、摘要、主题沉淀、成稿和发布包装已经有较强基础，但对外正式发布、状态追踪、失败重试和审批审计仍需要继续补齐。

## 1. 业务目标

- 把链接、本地文件、主题要求等输入稳定转为可复用的知识资产
- 让内容从“来源”继续推进到“主题”“成稿”“发布包装”
- 让同一份内容尽量产出分层结果，便于后续复用、改写和发布
- 让后续 agent 能按照固定结构继续接手，而不是每次从头判断

本次闭环不包含：

- 全自动跨平台正式发布
- 无人工确认的品牌定稿
- 需要高风险登录态和不可逆操作的自动执行

## 2. 当前现状

- 仓库已经有稳定目录结构，覆盖来源、笔记、输出、模板、工作流、技能和系统说明
- 已有自动分类入库脚本 `scripts/intake_source.py`
- 已有来源摘要、本地发布包装、视觉资产和 Markdown 预览等 skill
- 已有多份系统文档，能支撑 agent 按约定执行
- 当前更强的是“文件系统驱动工作流”，相对较弱的是“状态机、队列、审计日志、统一任务面板”

当前主要卡点：

- 任务状态更多依赖文件散落和上下文记忆，缺少统一状态追踪
- 多来源聚合、失败重试、重复去重还没有形成统一执行器
- 对外发布前的人工审核点没有标准化成显式状态流转
- 平台分发动作仍主要停留在人工或半人工操作

## 3. 输入层

| 输入项 | 来源 | 格式 | 是否稳定 | 备注 |
|---|---|---|---|---|
| 外部文章链接 | 用户输入 / 网络来源 | URL | 高 | 适合先自动入库和生成来源笔记 |
| 视频链接 | 用户输入 / 平台页面 | URL | 中 | 是否能稳定拆解，取决于字幕和页面可读性 |
| PDF | 本地文件 / 下载文件 | 文件路径 | 高 | 适合先转 Markdown 再提炼 |
| Word / PPT / Excel / CSV | 本地文件 | 文件路径 | 中 | 结构差异较大，通常需要先分类再处理 |
| 主题要求 | 用户输入 | 自然语言 | 中 | 适合生成主题笔记或草稿，但需要更强人工判断 |
| 已有草稿 | 仓库现有 Markdown | 文件路径 | 高 | 适合继续进入发布包装和预览流程 |

## 4. 动作层

| 动作 | 当前执行方式 | 是否已脚本化 | 风险 | 备注 |
|---|---|---|---|---|
| 来源类型判断 | agent + `intake_source.py` | 是 | 低 | 当前已具备稳定入口 |
| 自动分类入库 | `intake_source.py` | 是 | 低 | 适合做为闭环第一步 |
| 来源摘要与结构化拆解 | `source-auto-summary` skill + agent | 部分 | 低 | 已有稳定方向，但仍需持续验证 |
| 主题笔记归并 | agent 根据工作流执行 | 否 | 中 | 缺少统一脚本和去重机制 |
| 文章草稿生成 | agent 根据主题笔记生成 | 否 | 中 | 可执行，但质量仍依赖主题笔记完整度 |
| 发布包装 | `article-publish-kit` skill | 部分 | 中 | 已形成稳定流程，适合半自动闭环 |
| 视觉资产生成 | `article-visual-assets` skill | 部分 | 中 | 已有校验机制，但仍需人工确认视觉质量 |
| Markdown 发布预览 | `serve_markdown_publish_preview.js` 及相关脚本 | 是 | 低 | 适合发布前校验 |
| 对外正式发布 | 人工执行 | 否 | 高 | 当前不建议直接自动化 |

## 5. 反馈层

| 反馈信号 | 如何获取 | 是否可量化 | 验收标准 |
|---|---|---|---|
| 来源笔记已生成 | 文件是否存在于目标目录 | 是 | 文件已创建，模板结构完整 |
| 摘要与关键信息已写入 | 检查 Markdown 内容 | 部分 | 包含摘要、事实、观点、判断等关键段落 |
| 主题笔记已建立 | 文件是否存在且双链齐全 | 是 | 已有主题定义、核心观点、相关来源 |
| 草稿已生成 | `outputs/drafts` 中是否有对应文件 | 是 | 已形成可读的工作稿 |
| 发布包装已补齐 | 资产和文档是否生成 | 是 | 正文、建议、视觉资产、传播物料达到最小集合 |
| 预览已通过 | 本地预览和校验脚本结果 | 部分 | 无明显布局错误、图片路径和复制效果可用 |
| 人工审核通过 | 人工确认 | 否 | 标记为可发布或进入下一阶段 |

## 6. 判断规则

- 成功：
  - 已完成当前阶段要求的产物
  - 下游所需输入已经补齐
  - 没有明显事实风险、版式风险或结构缺失
- 失败：
  - 文件未生成
  - 结构严重不完整
  - 关键脚本报错且无法自动恢复
  - 关键输入缺失，无法继续推进
- 可重试：
  - 预览服务异常
  - SVG 校验未通过
  - 来源处理因格式问题失败
- 必须人工介入：
  - 主题边界不清
  - 发布口径需要决策
  - 事实准确性存在明显不确定性
  - 对外正式发布前确认

## 7. 状态机设计

建议当前仓库的主状态机为：

`new -> processing -> blocked -> ready_for_review -> approved -> published`

### 状态说明

| 状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| `new` | 新输入待处理 | 新链接、文件、主题或草稿进入仓库 | 已开始入库或分析 |
| `processing` | 正在处理 | agent 或脚本已接管任务 | 形成中间产物或遇到阻塞 |
| `blocked` | 被输入或异常阻塞 | 缺关键输入、脚本失败、来源无法读取 | 问题解决后回到 `processing` |
| `ready_for_review` | 已形成可审稿结果 | 已有来源笔记、主题笔记、草稿或发布包初版 | 人工完成审阅 |
| `approved` | 允许进入发布或归档 | 人工确认内容与资产可用 | 发布或归档完成 |
| `published` | 已完成对外分发 | 已在目标渠道发布或确认已分发 | 进入复盘或沉淀 |

## 8. 人工闸门

- 主题笔记是否足够稳定，通常需要人工判断是否值得继续成稿
- 发布版正文在正式对外发送前，应进行人工审核
- 视觉资产在对外平台使用前，应人工确认版式与品牌口径
- 平台分发、复制到飞书、公众号或其他渠道时，应保留人工最终确认

## 9. 异常处理

| 异常类型 | 是否可重试 | 处理方式 | 接管人 |
|---|---|---|---|
| 来源无法读取 | 是 | 更换读取方式、补一手来源、人工提供材料 | 执行 agent / 人工 |
| 自动入库失败 | 是 | 检查路径、命名和脚本输入 | 执行 agent |
| 摘要结构缺失 | 是 | 重新生成并按模板补齐 | 执行 agent |
| SVG 越界或重叠 | 是 | 修 SVG 或切换 `Satori` 模板 | 执行 agent |
| 预览服务异常 | 是 | 重启预览服务并重新验证 | 执行 agent |
| 对外发布判断不清 | 否 | 转人工审核并等待结论 | 人工 |

## 10. 当前已能跑通的部分

- 输入侧已经有稳定目录和来源类型划分
- 自动入库已有明确脚本入口
- 来源摘要与发布包装已经有可复用 skill
- 视觉资产已有 SVG 优先、PNG 验证和布局检查的流程
- 发布前的 Markdown 预览已有本地服务支撑
- 文档、模板、工作流和技能已经形成较好的文件化协作基础

## 11. 还缺的基础设施

- 统一任务状态表或任务索引页
- 更明确的阶段状态回写机制
- 多来源合并与去重规则
- 系统化失败重试与审计日志
- 发布前审核清单的标准化记录
- 对外发布后的结果回收与复盘机制

## 12. 推荐落地路径

### 第一阶段

- 先把 `new / processing / ready_for_review / approved / published` 状态写进相关文档或 frontmatter
- 先让来源、主题、草稿、发布包都能显式标注所处阶段
- 先统一用现有模板和技能把中间产物补齐

### 第二阶段

- 增加任务索引页或状态总览页
- 为重复步骤补统一脚本或固定命令入口
- 明确异常类型与重试策略

### 第三阶段

- 再评估是否把“业务闭环设计”沉淀成独立 skill
- 再补更强的发布审核、审计日志和外部平台回写机制
- 条件成熟后再考虑部分渠道级自动发布
