# 用户本人写作指纹 · 随笔 mode（_self-essay）

跟 [_self.md](/Users/apple/Desktop/project/document/99_System/voice-references/_self.md) 是同一作者的两种 mode，不是替代关系。
方法论 mode 写"读者照着干"的 SOP，随笔 mode 写"读者改变判断"的观察 / 复盘 / 评测。

## ⚠️ 样本说明（必读）

- **样本极少**：用户 blog.workmn.com 历史 ~60 篇里几乎全是方法论 / 教程 / bug 速记，**没有真正的"随笔型"独立成篇**
- **本基线主要从 12-components 这一篇 essay 反推**，副样本是 OpenSpec 篇前 3 段（少数带"我读完"反思口吻的段落）
- 因此 essay mode 基线**置信度低、采样面窄**，应在用户后续写出 3-5 篇随笔后回来重写，不要当作"已稳定"
- **判断依据足够明确的部分**：跟方法论 mode 的差异点（无 TL;DR、无 FAQ、无三件套、章节标题是观点句）
- **不够明确的部分**：起手定式的稳定模板、收尾结构（目前只有 1 个样本，无法说"招牌做法"）

## 量化基线（双样本对照）

**核心反差点（essay vs 方法论）**：

1. **没有 SOP 骨架**——essay mode 直接被砍掉的：TL;DR / 适用读者 / 问题定义 / 目标 / 通过标准 / 最小可复现验证 / FAQ 兜底
2. **章节标题从功能段名变成观点句**——"会话短命，磁盘长命"取代"问题定义（Problem Statement）"
3. **"——"破折号大量出现**——方法论 mode 几乎只用括号，essay mode 用破折号做"句中断点 + 加重 + 自答"
4. **"我读完 / 我数了一遍 / 我的结论"动词链高频**——essay 强调"我亲自看过"，方法论强调"流程应该这样"
5. **代码块密度更高**——essay 的代码是"证据"（"看，作者真的写了 0 个 if-else"），方法论的代码是"示例"（"该这么用"）
6. **配图可以有 1 张**——essay 模式下，认知图 / 架构图作为"读者快速抓重点"的入口，可以出现；方法论 mode 默认 0 张

## A 结构层 5 条指纹（essay mode）

### A1. 起手定式：「读者代入 + 钩子事实 + 反共识断言」三段

观察评测体（12-components 主样本）：

- 第 1 段："你天天用 Claude Code，可能从来没想过它内部到底有几个组件。"——读者代入
- 第 2 段：报出来源（GitHub 仓库 + 字数 + 自己读过）——钩子事实，建立可信度
- 第 3 段：直接抛反共识结论（"外界普遍流传的那张'12 组件平铺图'是错的"）——反判

复盘体（OpenSpec 前 3 段）：

- 第 1 段："我最近越来越明确一件事：…最贵的成本已经不是'写'，而是**返工**。"——个人观察起手
- 第 2 段：自我反思（"以前我会把这个问题归结为 prompt 不够准…但我后来发现…"）
- 第 3 段：引出新做法（"也正是因为这个原因，我开始认真研究 OpenSpec"）

**两者共同特征**：起手 3 段都不进入 TL;DR / 问题定义这种"工程文档动作"，而是先建立一个"读者愿意继续读下去"的钩子。

→ sub-agent 写 essay 时，**严禁开篇直接放 TL;DR**。先用 3 段把读者拉进来。

### A2. 章节标题是观点句，不是功能段名

12-components 里 9 个 `##` 全是判断句：

- "误读 1，12 个组件不是并列的，是 2 层架构"
- "内层 5 组件，循环不动，只往工具箱加东西"
- "关键设计，SKILL.md 的两层加载"
- "外层 7 组件，磁盘才是真正的协调骨架"
- "Agency 来自训练，不来自编排"
- "工具壳易迭代，架构判断会沉淀"

跟方法论 mode 的"问题定义 / 目标与非目标 / 实施步骤 / 通过标准"段名形成强对照。

→ sub-agent 起每个 `##` 时，问自己：这是一句"我的判断"还是一个"工程文档段名"？essay mode 必须是前者。

### A3. 没有"工程文档三件套"，但保留代码块作为证据

essay mode 不写：

- "通过标准（建议）"
- "最小可复现验证（示例）"
- "不通过先查"

但保留**比方法论 mode 更密集的代码块**——12-components 6 块代码 vs OpenSpec 3 块。代码块的角色变了：

- 方法论 mode：代码 = "你应该这样写"（指导性）
- essay mode：代码 = "看，证据在这"（证据性，用来支撑前文断言）

例：12-components 第 2 节贴 30 行 agent_loop 代码，紧跟一句"这就是 Claude Code 内核的全部"——代码是用来"亲自验证我的论点"的。

### A4. 中段插一张对照表 / 一张图（可选但稳定出现）

12-components 出现 1 张对照表（"内层 vs 外层 对照"）+ 1 张认知图（架构分层 PNG）。

essay mode 比方法论 mode**更鼓励视觉资产出现 1 个**——因为读者读 essay 是为了"快速形成新判断"，一张对照表 / 认知图能压缩 200 字解释。

但**最多 1 张图 + 1 张表**，不是堆。差异跟卡兹克"1/100 字配图密度"明确不同——essay mode 仍然是文字本位，图只是"判断的浓缩"。

### A5. 收尾用"心法句 / 一句话留下"+ 引出下一步阅读

12-components 收尾（最后两段）：

```
读完 12 章留下来的一句心法是，当 agent 需要"长出新行为"时，
先问自己——这是该加一个工具，还是该改一层协调骨架。十次有九次，答案是前者。
```

外加一句"`claw0` 是下一个值得读的仓库"——给读者下一个钩子。

OpenSpec 篇收尾（带温度的复盘段）：

> 对我来说，这不是一套'更正式的文档方法'，而是一次协作方式的升级。

**特征**：

- 不写 FAQ 兜底（方法论 mode 的招牌，essay 直接砍）
- 不写"4 字口号"（卡兹克的招牌，essay 也不用）
- 而是用"一句心法 / 一个对仗判断" + "下一个钩子"

→ sub-agent 写 essay 收尾时，**先想这一句心法能不能被读者收藏并转述**。如果能，留；如果是空话，删。

## B 语言层 6 条指纹（essay mode）

### B1. 第二人称"你"作为"读者引导者"高频出现

12-components 17 处"你"，几乎全是引导动作：

- "你天天用 Claude Code，可能从来没想过…"
- "如果你在自己项目里照搬…"
- "如果你做的是'领域内 single-purpose agent'，内层 5 组件够了"
- "如果你 agent 需要跨进程通信，先想想能不能…"

**特征**：essay mode 比方法论 mode 的"你"用得更口语 / 更读者代入，但**仍然不是营销煽动**（不会出现"你也可以！""相信我"）。

→ 写 essay 时，"你"应主导，"我"应克制（除非是复盘体起源篇，那种情况下"我"反过来主导）。

### B2. "——"破折号大量取代括号注释（与方法论 mode 反差最大）

12-components 18 次"——"。例：

- "读完代码我的结论不一样——**前 5 章和后 7 章是两个不同物种**。"
- "这条判断在源码里反复出现——s07 的 TaskManager 故意没用内存对象…"
- "为什么后半段全是磁盘。读 s07 的 docstring 我才反应过来——**会话上下文随时会被 context compact 抹掉**…"

破折号的角色：

- **句中加重**：把后半句变成重音
- **自问自答**：先抛问题，破折号后给答案
- **替代"因为"/"是因为"**：节奏更紧

→ essay mode 写作 sub-agent **应主动用"——"打节奏**，方法论 mode 的"括号注释（说明）"风格在 essay 里要弱化。

### B3. "我读完 / 我数了一遍 / 我抄了 / 我看到 / 我反应过来"动词链

12-components 8+ 次：

- "我读完 12 章源码…读完反而发现"
- "12 章源码里我数了一遍——0 处用 if-else 替模型做决策"
- "读完 12 章我抄了 3 个模式到自己手边"
- "读 s07 的 docstring 我才反应过来"

**特征**：essay mode 的"我"不是"我认为 / 我建议"那种意见性，而是**"我亲自做了 X 才知道 Y"的动作链**——强调一手经验，不是空议论。

→ sub-agent 写 essay 时，**至少出现 3-5 处"我读 / 我看 / 我数 / 我抄"型动词链**。如果一篇 essay 全是"我认为"，会显得空。

### B4. 反共识断言句（"外界都说 X，但 X 是错的"）

12-components 全篇围绕一个反共识：

- "外界普遍流传的那张'12 组件平铺图'是错的"
- "误读 1，12 个组件不是并列的"

OpenSpec 前 3 段同样：

- "以前我会把这个问题归结为 prompt 不够准…但我后来发现，很多问题根本不是'提示词技巧'能解决的"

→ essay mode **应有一个明确的"我反对的判断"**。如果文章读完没看到作者反对什么，essay 没立住。

### B5. "把 X 当作 / 当成 Y" 重定义句式（继承方法论 mode）

essay 仍然偶尔出现，但密度明显比方法论 mode 低（12-components 1 次 vs OpenSpec 全文 3-5 次）：

- "把它**当作**一条**反馈闭环**"（方法论里频繁）
- 在 essay 里出现频次降低，但**保留作为"概念升级"动作**

### B6. "不是 X 而是 Y" 在 essay mode 里反而用得少

意外发现：**12-components 全文 0 次"不是 X 而是 Y"**。OpenSpec 前 3 段也是 0 次（虽然全文 8+ 次，但都在中后段方法论部分）。

**推测原因**：essay 是"我亲眼看到 + 我的判断"，反共识用具体动词（"读完反而发现"、"我数了一遍"）来表达；方法论 mode 是"概念定义 + 概念修正"，需要"不是 X 而是 Y"的对仗句式来辨析定义。

→ essay mode 里**不刻意追求"不是 X 而是 Y"**。如果出现也行，但不该把它当招牌。

> essay mode = 砍掉 SOP 骨架 + 章节标题改观点句 + "——"破折号打节奏 + "我读完 / 我数了一遍"动词链做证据 + 收尾用一句心法和下一个钩子

## essay mode 的两个子型

样本太少不能下定论，但 12-components 和 OpenSpec 复盘段已经表现出**两种 essay 子型**：

### 子型 a：观察评测体（12-components 是样本）

- "我"克制（5-10 次），"你"主导（15-25 次）
- 围绕一个**外部对象**（仓库 / 工具 / 文章 / 现象）展开
- 章节标题是反共识判断
- 大量代码 / 截图 / 数据作证据
- 收尾给读者下一个钩子

适用：技术评测、工具拆解、源码精读、行业现象观察

### 子型 b：复盘体起源篇（OpenSpec 前 3 段是样本）

- "我"爆表（70+ 次），"你"几乎不出现
- 围绕**自己的方法论起源**（"我为什么开始用 X"）
- 章节标题仍可能是 SOP 段名（OpenSpec 中后段就是这样混着）
- 第一人称密度 1/40 字，是用户唯一会出现"温度"的写法
- 收尾用对仗温度句

适用：方法论的"诞生故事"、个人转型复盘、回头看自己一年的做法

→ sub-agent 选 essay mode 时，再判一次是哪个子型：评测/拆解 → 子型 a；起源/复盘 → 子型 b。

## 写作 sub-agent 怎么用这份指纹

### 默认对齐（写用户名义的随笔、技术评测、源码精读、现象观察、起源复盘）

1. **结构强制**：5-9 个章节，**每个章节标题必须是观点句，不是 SOP 段名**
2. **起手强制**：3 段拉读者进场（不许直接放 TL;DR）
3. **三件套禁止**：不写 TL;DR、不写适用读者、不写问题定义、不写通过标准、不写 FAQ
4. **节奏强制**：全篇至少 10 处以上"——"破折号，主要用在"句中加重"和"自问自答"
5. **证据强制**：至少出现 3-5 处"我读完 / 我数了一遍 / 我看到 / 我抄了"动词链
6. **反共识强制**：全文应有 1 个明确反对的判断，且在前 3 段抛出
7. **收尾强制**：用一句心法 + 引出下一个钩子，**禁止 FAQ 兜底**
8. **代码作证据，不作示例**：贴代码后紧跟一句"看，作者真的写了 0 个 if-else"型论断
9. **配图最多 1 张**（认知图 / 架构图），表格最多 1 张（核心对照）
10. **保留共用**：B4 词典词（口径 / 信号 / 护栏 / 工件 / 真相 / 闭环）继续用，中英术语继续裸用

### 不对齐（破圈到公众号 / 小红书时）

按 [khazix.md](/Users/apple/Desktop/project/document/99_System/voice-references/khazix.md) 重写：碎段、灌情绪、配图密度上来。**这种情况下临时关掉本指纹**。

### essay vs 方法论 mode 的判定（接给 _index.md）

详见 [_index.md](/Users/apple/Desktop/project/document/99_System/voice-references/_index.md) 新增的"用户双 mode 自动判定"段。简版规则：

- 章节标题里出现「TL;DR / 适用读者 / 通过标准 / 最小可复现验证 / FAQ」→ 选 _self.md（方法论）
- 关键词触发「我读完 / 我数了一遍 / 我看到 / 误读 / 评测 / 反共识 / 起源 / 复盘 / 现象」→ 选 _self-essay.md（随笔）
- 都不命中：默认 _self.md（更稳）

## 反例 / 警惕

- **别套了 essay 形式忘了反共识**：essay mode 的灵魂是"我反对什么"。如果一篇 essay 读完读者不知道作者反对什么，结构再像也不像 essay
- **别把"——"当万能装饰**：破折号要承担节奏，乱撒会变碎句
- **别让"我读完 / 我数"变成口头禅**：这些动词链要紧跟一个具体的"看到了什么"，否则变成空喊"我读过哦"
- **别把方法论 mode 的 SOP 骨架"减半"当成 essay**：essay 不是"压缩版方法论"，是另一种结构。砍掉 TL;DR 和 FAQ 之后必须**用观点句重组章节**，不是在 SOP 段名里挑几个留下
- **样本不足提醒**：本基线只有 1.5 个样本支撑，跑 voice-match 出现明显不像的字段时，**优先怀疑本基线，不是文章**。等用户写到 3-5 篇随笔时回来重写本文件
