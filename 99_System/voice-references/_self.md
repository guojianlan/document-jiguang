## A 结构层 6 条指纹

### A1. 模板化的"方法论 8 段式"

成熟期方法论文几乎复用同一个骨架，**先扔结论再展开论证**：

```
1. 起手 2-3 段（场景痛点 + 错误做法 + 我的判断）
2. TL;DR（30 秒讲清楚）
3. 适用读者与前置知识（适合 / 不适合 / 前置）
4. 问题定义（Problem Statement，单段 blockquote）
5. 目标与非目标（Goals / Non-goals）
6. 方案与权衡（核心，1.X / 2.X / 3.X 编号小节）
7. 实现要点 / 真实链路（带步骤编号 1-8）
8. 实践步骤（Step-by-step，编号清单）
9. 指标与验证（含"通过标准"+"最小可复现验证"+"不通过先查"）
10. 常见坑与规避（Pitfalls）
11. FAQ（5-8 条，Q/A 标题层级）
```

**用户本人 = 工程文档的 SOP 化作者**。这套结构在 OpenSpec / 可观测性 / 导出链路 / BFF / SEO Meta 5 篇方法论文中**全部出现**，不是巧合，是 SOP。

我们的写作 sub-agent 处理用户本人风格时，**应优先复用这套骨架**，不要自创结构。

### A2. 起手三句话定式："场景描述 + 错误做法举例 + 我的反判"

用户起手几乎不用"昨天"、"最近"这类时间锚点，而用一种**结构化定义场景**的开场：

- OpenSpec："我最近越来越明确一件事：AI 写代码这件事，最贵的成本已经不是'写'，而是**返工**。"
- 可观测性："很多团队做可观测性，第一步是'接入一个平台'，第二步是'看一眼报错'，然后就没有然后了。"
- BFF："入口一多，路由就不再是'写几条规则'的小事了。"
- SEO Meta："当一个站点同时包含获客页与产品功能页…SEO Meta 往往会以一种'很自然但很危险'的方式散装化。"

**核心句式**：「[场景条件] + [常见做法] + [但是会失败]」三段。这是"问题驱动"的起手，跟卡兹克的"故事驱动"起手相反。

### A3. TL;DR 用 5 个加粗 bullet 点收束全文

每篇 TL;DR 必有 4-5 个 bullet，每个 bullet 用 `**词：**` 加粗起头：

- `**核心做法：**…`
- `**问题：**…`
- `**方案：**…`
- `**验证：**…`
- `**底线：**…`

**形式上极度稳定**——所有方法论文都用这套句法。读者哪怕只看 TL;DR 也能拿走核心结论。

### A4. 用 blockquote 包"问题定义"

每篇都有：

```
## 问题定义（Problem Statement）

> [一段 100-200 字的目标陈述，单段，不分行]
```

这是一个**正式的工程文档动作**——把"我要解决什么"形式化。这条指纹在文章中段出现 100% 命中，不能删。

### A5. "通过标准 / 最小可复现验证 / 不通过先查" 三件套作为收尾结构

成熟期方法论文几乎都在文末有：

- **通过标准（建议）**：bullet 列出"这条流程跑出什么样才算成功"
- **最小可复现验证（示例）**：贴 3-5 行可复制 `curl` / `bash` 命令
- **不通过先查**：bullet 列出 4-6 个常见失败原因

**这是用户最强的工程指纹**——把方法论"留作业"了，读者拿到不是空话，而是可以立刻验证的步骤。卡兹克完全没有这种结构。

### A6. FAQ 收尾，不写"金句"

用户的方法论文几乎不写"金句结尾"，而是 5-8 个 FAQ 兜底，每条 100-200 字。**FAQ 取代了情绪收尾**。

唯一例外是 OpenSpec 篇有一段类似收尾的话：

> 对我来说，这不是一套'更正式的文档方法'，而是一次协作方式的升级。我希望以后我和 AI 的关系，不再是'我说一点，你猜一点'，而是'我们先把这次变更讲清楚，再一起把它做对'。

**特征**：长句、"X 而是 Y" 句式、对仗、不堆短句。**温度有，但克制**。

## B 语言层 8 条指纹

### B1. "不是 X，而是 Y" 反判句式（高频）

样本里反复出现：

- "**不是**让 AI 更敢写，**而是**想让它更少写偏。"
- "**不是**'有没有日志'，**而是**'发生了什么、影响谁、为什么、下一步做什么'。"
- "**不是**在加文档，**而是**在补'协作协议'。"
- "**不是**'更聪明地聊天'，**而是**'更稳定地复用某类能力'。"
- "**不是**…**而是**…" 在 OpenSpec 篇里出现 8+ 次。

**警告**：仓库 critique 规则把"而是"列为去 AI 味硬规则之一。但用户本人**就是大量用这个句式**。

→ **这条要单独标记**：用户本人风格 ≠ critique 默认规则。当目标平台是用户本人博客 / 掘金类工程社区时，"不是 X 而是 Y" 应**保留**；当目标是公众号 / 小红书时，按 critique 规则改写。

### B2. "把 X 当作 / 当成 Y" 重定义句式

把抽象问题装进具体容器：

- "把它**当作**一条**反馈闭环**"
- "把 BFF **当成**'跨切面能力层'"
- "把路由治理**当作**一个'系统能力'"
- "**不再把** prompt **当成**唯一入口，**而是**把规格工件**当成**真正的协作入口"

跟 B1 经常组合使用。**这是用户处理"概念升级"的招牌动作**。

### B3. 物理化动词（中等密度，工程化变体）

不是卡兹克的"摁在地上打"那种生猛，而是**工程化的物理化**：

- "**关进笼子**"（迁移风险）
- "**钉住**"（边界）
- "**散落 / 散装**"（规则分散）
- "**叠前缀**"（locale URL bug）
- "**跑版 / 乱序**"（排版失败）
- "**信号会不稳定**"
- "**烘焙进位图**"（透明度处理）
- "**打架**"（旧新规则同时开启）

**特征**：把抽象的工程概念物理化、容器化，但保留专业语境。读者一看就知道是写代码的人写的，不会以为是 PM。

### B4. "口径" / "信号" / "护栏" / "工件" / "真相" 等抽象名词词典

用户喜欢用一些抽象但有重量的名词：

- **口径**："SEO 口径一致" / "head 与 sitemap 同口径"
- **信号**："SEO 信号" / "三类信号" / "暴露信号"
- **护栏**："变成护栏" / "回归护栏"
- **工件**："稳定工件" / "中间工件" / "规格工件"
- **真相**："单一真相路由表" / "新的系统真相" / "当前真相"
- **闭环**："反馈闭环" / "可观测闭环" / "三件套闭环"

这套词典出现在每篇方法论文里。**这是用户的"个人词典"**——sub-agent 复用时这些词应**优先选用**而不是同义词替换。

### B5. 第二人称"你"作为"读者协作者"称呼，不是营销话术

**关键区分**：用户用"你"不是营销煽动（"你也可以！"），而是工程协作（"你现在该…"）。例：

- "你要回答的不是…而是…"
- "你至少需要…"
- "你会发现，'统一入口'并不是…"
- "如果你只能做一件事：先…"

**比"我"用得多**（成熟期方法论文里）。**早期教程篇（SaaS / 双 Token）和 2025-11 OpenSpec 个人复盘篇里反过来——"我"占主导**。

→ **场域规则**：写"个人复盘 / 方法论起源"时，**第一人称密度拉高**；写"工程实操 / 系统设计"时，**第二人称密度拉高**。

### B6. 中英术语混排，不翻译，不加引号

成熟期方法论文里大量直写英文：

- canonical、alternates、noindex、hreflang、SSR、SSOT、TTFB、p95、MTTD、MTTR、IR、BiDi、RTL、shaping、fontkit、subset、proposal、specs、tasks、archive、proxy_buffer_size、Crash-free sessions、autocapture…

**特征**：术语**不翻译、不解释、不加""**——默认读者是同行。

→ 写用户风格的稿时，**英文术语该裸用就裸用**，不要做"贴心的中文注解"。

### B7. 标点偏好：英文括号包注释、破折号不用、句号不省

- "（30 秒讲清楚）" / "（建议）" / "（可选但很有用）" / "（避免循环依赖）"——**括号注释极多**
- 极少用破折号 `——`，更倾向用括号或冒号
- 几乎不用感叹号、波浪号、emoji（卡兹克爱用的"～""。。。。。。"完全没有）
- 偶尔用 ASCII 箭头 `→`（"clicked → completed → failed"）做流程

**结论**：标点上**克制、冷静、工程化**。这是跟卡兹克最直观的视觉差。

### B8. 自创词 / 命名能力强

用户经常自己造短语命名一个工程概念：

- "**统一事件封装**" / "**event envelope**"（自起的术语）
- "**导出链路三件套**"
- "**可关联 / 可验收 / 可控**"（三件套形式）
- "**可定位 / 可验证 / 可治理**"
- "**可一致性 / 可迁移 / 可验证 / 可覆盖**"（"可 X" 系列）
- "**SOP / SSOT** 类大写缩写**自己造**"
- "**漏写 / 写错 / 不一致**"（事故三类）

**模式**：喜欢 3 个并列的"可 X / X 化"短语作为方法概括。**这是用户独有的"概念命名能力"**——卡兹克是"绰号能力"（小龙虾、整活），用户是"工程命名能力"（三件套、统一封装、单一真相）。

## 写作 sub-agent 怎么用这份指纹

### 默认对齐（写用户名义的稿、用户博客发的稿）

1. **结构强制**：套用 A1 的"方法论 8 段式"骨架。少于 5 个章节的文章直接判返工。
2. **起手强制**："场景定义 + 常见做法 + 反判"三句式开头，**严禁**"昨天"、"最近"、工位段子起手。
3. **TL;DR 强制**：4-5 个加粗 bullet 起头。
4. **三件套收尾**：每篇必有"通过标准 + 最小可复现验证 + 不通过先查"，可选用 FAQ 替代部分。
5. **词典对齐**：B4 词典里的词（口径/信号/护栏/工件/真相/闭环）优先用，不要替换为同义词。
6. **句式可保留**："不是 X，而是 Y" 在用户风格下**保留**，不按 critique 默认规则改写。
7. **配图 0 张默认**：用户博客几乎无配图，sub-agent 不要主动加图。

### 不对齐（用户希望"破圈"，目标是公众号 / 小红书时）

按 [khazix.md](/Users/apple/Desktop/project/document/99_System/voice-references/khazix.md) 重写：碎段 + 灌情绪 + 配图密度上来 + 收尾三段式。**这种情况下临时关掉本指纹**。

### 中间态（写"我的方法论起源"复盘类，OpenSpec 篇就是样本）

A 结构层照本指纹（10 章 + TL;DR + FAQ），**B 语言层第一人称密度拉到 1/40 字**——这是用户唯一会出现"温度"的场景。

---

## 反例 / 警惕

- **别套了结构忘了思考**：用户的 8 段式不是排版皮，每段都有真内容（"通过标准"必有可执行 bullet，"最小可复现验证"必有可复制命令）。空套结构 = 仿不了用户。
- **别把"克制"写成"冷漠"**：用户文章读起来"专业但温和"，靠的是 B1 的"不是 X 而是 Y" + B2 的"把 X 当作 Y"，**句式自带温度**。如果只剩干说明书，就走味了。
- **别突然冒出 emoji / 短句独立成段**：用户从不这么写。一旦出现就不像他。
- **别拿早期 Nuxt 系列（2020）当样本**：那批是流水账，不代表当前。
- **早期 SaaS 教程系列（2024-06）跟方法论文风格也不同**：教程体更"步骤展示+代码贴片"，方法论体才是当前主笔风格。sub-agent 优先对齐方法论体。
