# 歸藏写作指纹（A 结构 + B 语言）

3 篇样本：自研工具发布短通告 / 第三方工具长测评 / 设计方法论自述 各 1 篇，覆盖他三类主体内容。

## 量化基线

| 维度 | 篇 1 自研发布 | 篇 2 工具测评 | 篇 3 设计自述 | 卡兹克对照 |
|---|---|---|---|---|
| 字数 | 1678 | 4613 | 2694 | 2536-3299 |
| 段数 | 33 | 141 | 104 | 106-137 |
| **段均字数** | **50.8** | **32.7** | **25.9** | **23.9-27.2** |
| 配图 | 4 | 24 | 10 | 20-31 |
| "我"次数 | 8 | 46 | 19 | 47-62 |
| "你"次数 | 10 | 52 | 11 | 15-58 |
| 短句独立成段（≤12字）| 4 | 9 | 30 | 多 |
| 显式分章节 | 0 | 0 | 6 | 0-12 |

**关键观察**：
- 段均字数 26-50，比卡兹克的 24 略长，但比一般博客（80-100）短得多。**短段密集**这点跟卡兹克在同一档。
- 第一人称密度比卡兹克明显低（比如自研工具发布篇 8 个"我" / 1953 字，1/200 字；卡兹克 1/50 字）。**作者在场感弱一档**，更像"产品经理介绍产品"，不像"老哥跟你聊"。
- 配图密度 1 张 / 269 字（取 3 篇加总），比卡兹克的 1 张 / 100 字稀一倍。
- 设计自述篇短句独立成段 30 个（比例最高），是他写"方法论 / 价值判断"段时的招牌节奏。

## A 结构层 6 条指纹

### A1. 内容前置交代型开场，不做钩子

3 篇都直接说"我在做什么 / 这个工具是什么"，**不绕、不戏剧化、不上来挂悬念**。
- "我这几天都在用 Vibe Coding 做我的 Agent 客户端 Codepilot。"
- "我现在的工作已经离不开 AI 浏览器了。"
- "上周被李继刚老师邀请去做了场私享会，关于 AI 和组织的访谈。"

跟卡兹克的"昨天，X 上线了" + 时间锚定 + 戏剧化情绪开场完全相反。**歸藏的开场是"信息直给"，把读者快速带进上下文**，不靠钩子留人。

### A2. 平行案例 / 平行字段做主体结构

工具测评和设计自述都用"案例 1 / 2 / 3 / 4 / 5" 或"5 套主题 / 6 个问题 / 三件事"的并列结构。**一篇文章 = 一份并列清单 + 短结尾**，结构感非常强，类似产品文档或评测报告。

样例：
- Tabbit 篇 5 个案例平铺
- PPT Skill 篇先列"它长什么样"4 个特征，再列"它能做什么"10 + 5 + 1 个交互
- "先跟 AI 说清这 6 件事" 直接编号 1-6

读者扫一眼能数清楚有几个点。**结构确定性 > 情绪起伏**。

### A3. 三段微章节切换

短文章里也用"## 它长什么样 / ## 它能做什么 / ## 怎么用 / ## 为什么长成这样 / ## 写在最后" 这种 2-4 字的小标题切块。**章节都是疑问句或动词短语**，不是名词式的"功能介绍 / 使用方法"。

跟卡兹克"教程类才分章，反思类全流式"不同——歸藏几乎所有类型都倾向于"轻分章"，让结构托住读者，不让节奏带读者。

### A4. 收尾三件套：复述价值 + 行动呼吁 + 转发请求

3 篇结尾几乎是同一个模板：
1. 一段平静总结（"也正因为这样，我才敢把它开源" / "改变使用习惯，才能发挥 Tabbit 的价值"）
2. 链接 / 安装命令（"https://github.com/op7418/..." / "npx skills add ..."）
3. "如果觉得今天的内容对你有帮助的话，可以帮我点个赞或者转发给需要的朋友们"

**没有金句、没有口号、没有四字成语收尾**。比卡兹克"长 → 短 → 口号"温和得多，更像产品 release note 的口径。

### A5. 价值判断段单独成段做强调

设计自述篇里大量出现独立成段的判断句，类似格言：
- "约束越严，风格越稳。 保护美学，比给用户自由更重要。"
- "没有节奏的 PPT 就是一沓 slide 堆成的 PDF。"
- "好看不是玄学，是一套可以拆解的决策。"

这些句子不靠情绪推动，靠**专业自信**支撑。设计师身份是他写这种判断句的底牌。**这一条只在他写"方法论 / 价值观"段时高频出现**，工具测评篇基本没有。

### A6. 信息密度 > 情绪密度

3 篇基本不"灌情绪"。Tabbit 篇 4613 字里，能找到的情绪句不超过 5 处（"非常方便" / "非常的全面" / "天哪" 这类温度都不高）。**他用信息量本身托住读完率**——读者来是为了"知道这个工具能干嘛"，不是为了"被带出小高潮"。

跟卡兹克 1 篇 5-6 处情绪锚点 + 收尾大爆点完全是两条路。

## B 语言层 8 条指纹

### B1. 第一人称密度 1 次/100-200 字

平均比卡兹克稀一倍。"我"出现是为了交代"我做了 / 我用了 / 我觉得"，**报告式在场**，不是"老哥式在场"。一个明显信号：歸藏写"我"经常带具体动作（"我这几天都在用"、"我让它抓取"），卡兹克写"我"经常带情绪（"我傻了"、"我惊了"）。

### B2. 短句独立成段做"价值判断"，不做"情绪爆点"

歸藏的短句独立成段，内容多是结论 / 原则 / 判断：
- "记忆不是档案。"（这是从小互文里借的对照，他自己是同型）
- "好看不是玄学。"
- "约束越严，风格越稳。"

跟卡兹克"恐怖如斯。 / 疯了吧。"那种情绪短句不同——**歸藏的短句是"压判断"，不是"灌情绪"**。

### B3. 列表 + 项目符号 + 编号高频

3 篇加起来出现"•" "1." "→" 数十次。任何能列项的内容他都列项。这是设计师 / 产品经理写法的强烈信号。**结构感来自排版**，不是来自情绪节奏。

### B4. 物理化动词稀少，多用"做、写、跑、装、拼"

物理化动词（卡兹克的"摁在地上打"、"刀刀砍"）在 3 篇中几乎为零。歸藏用动词偏向中性技术口径："拼出来、跑通、压进去、串起来"。**最物理化的也只是"我把它们一条条写进了 SKILL.md"**，远远没到卡兹克的强度。

### B5. 口语爆点词低频，绝不出现"YYDS / 狗东西 / 老子"

3 篇能找到的口语词：
- "拉满"（出现 1 次）
- "卡的要死"（出现 1 次）  
- "牛P"（小互的，不在歸藏样本里）

**亲和力不靠口语爆点**，靠"我这两天试了" "我自己用了一周后" 这类亲历感。

### B6. 中英混排 + 产品名 / 命令行直出

凡是涉及工具，必带原名 + 链接 / 命令：`Codepilot`、`Claude-to-IM-skill`、`npx skills add op7418/...`、`mcporter config add ...`。**不翻译、不科普**，默认读者听得懂术语。这是面向开发者 / 产品经理 / AI 工具圈的精准定位，不是普通公众号读者。

### B7. 设计行业术语作为身份标签

PPT Skill 篇是密集示范："衬线 / 非衬线 / 等宽"、"7:5 / 6:6 / 8:4 网格"、"hero 页 / non-hero 页"、"密度 / 节奏 / 呼吸感"。**这套术语是他和读者建立"我是十年设计师"信任的关键**。

非设计篇也偶尔露："非常优雅"、"易用"、"这套澄清流程"——价值判断词偏审美 / 结构 / 系统，不偏速度 / 强度 / 爽感。

### B8. 几乎不用"！"，问号也少

3 篇加起来叹号不超过 10 个（基本只在"哈哈哈"或引用别人话时出现）。问号集中在"先跟 AI 说清这 6 件事" 那种结构性提问。**句号是主调**。语气平、节奏稳。

跟卡兹克"！" + "。。。。。。" + "～" 的强标点节奏完全相反。

## 跟卡兹克对照表

| 维度 | 卡兹克 | 歸藏 |
|---|---|---|
| 段均字数 | 24 | 25-50 |
| 第一人称密度 | 1/50 字 | 1/100-200 字 |
| 开场套路 | "昨天，X" + 戏剧化 | "我现在 / 上周" + 直给 |
| 收尾节奏 | 长 → 短 → 口号 | 平静总结 + 链接 + 转发请求 |
| 情绪锚点 | 5-6 处 / 篇 | 0-1 处 / 篇 |
| 物理化动词 | 多（"摁在地上打"）| 极少（"压进去"已经是上限）|
| 口语爆点 | 多（YYDS / 狗东西）| 几乎无（"拉满 / 卡的要死" 是边界）|
| 配图密度 | 1 张/100 字 | 1 张/270 字 |
| 列表项目符号 | 极少 | 高频 |
| 标点节奏 | "！" + "。。。。。。" + "～" 多 | "。" 为主，几乎无强标点 |
| 章节切分 | 看类型决定（教程才分）| 几乎都轻分（2-4 字疑问句标题）|
| 价值判断句 | 含情绪（"恐怖如斯"）| 含审美（"约束越严，风格越稳"）|
| 风格定位 | 老哥跟你聊 | 设计师 + 产品经理双身份做技术分享 |

**结论**：两人都是公众号长文派，但走的是反方向：
- 卡兹克 = 高情绪密度 + 高配图密度 + 强口语 + 戏剧化开场，靠**情绪推读完率**
- 歸藏 = 高信息密度 + 强结构 + 中英混排 + 平静收尾，靠**结构和专业感推读完率**

## 定位标签

```
歸藏 = 设计审美派 × 自研工具开发者 × 列表型评测者
```

三重身份叠加：
- **设计审美派**：色彩 / 字体 / 网格 / 节奏术语随手就来，价值判断句带审美权威感
- **自研工具开发者**：项目名 / GitHub URL / 安装命令直出，对开发者友好
- **列表型评测者**：5 案例 / 6 问题 / 三层结构这种平铺并列结构是默认武器

## 什么场景调他

写下面这些类型的稿子，可以参考歸藏的指纹：

| 场景 | 该抄哪几条 |
|---|---|
| 工具评测 / 软件介绍 | A2 平行案例 + B3 列表 + A4 平静收尾 + B6 中英混排 |
| 自研产品发布 | A1 信息直给开场 + A3 轻分章 + A4 链接 + 转发请求 |
| 设计 / 方法论自述 | A5 价值判断段 + B2 短句压判断 + B7 行业术语 |
| 给开发者圈写技术心得 | B6 命令行直出 + A2 平行字段 + 全文不灌情绪 |

**反过来**——下面这些场景**不要**抄歸藏：
- 想冲点击 / 朋友圈传播：他的开场太平，不抓人
- 想写情感共鸣 / 个人成长：他的语言层没有情绪锚点工具
- 想吸引非技术圈读者：中英混排 + 命令行会劝退普通读者

## 改写示例

取我们之前的 12 components 文章原段：

> 原文：
> GitHub 上 shareAI-lab 的 `learn-claude-code` 把这件事掰得很清楚——12 节课，12 个递进的机制，每章在前一章基础上叠加一个东西。我读完 12 章源码（每章 100-800 行 Python，加起来 4000 多行），读完反而发现，外界普遍流传的那张"12 组件平铺图"是错的。

按歸藏风格重写：

> 我这两天读完了 shareAI-lab 的 `learn-claude-code`。
>
> 12 节课，每节 100-800 行 Python，加起来 4000 多行，把 Claude Code 的内部一层层拆开。
>
> 读完之后发现一件事：
>
> 外界普遍在传的那张"12 组件平铺图"，结构是错的。
>
> 它把 12 个组件画成 12 个并列方块，仿佛它们是同一层的兄弟。
>
> 但代码里它们不是。
>
> 真实的结构是 12 层依次叠加，每一层在前一层基础上加一个机制。
>
> 这一点，我觉得值得单独写一篇。

差异点（跟卡兹克版对比）：
- 没有"昨天熬夜"时间钩子（A1 信息直给）
- 没有"我傻了 / 真的，全是"情绪短句（B2 短句压判断而非情绪）
- 用"读完之后发现一件事"这种平铺式承转，不上戏剧化转折
- 收尾"我觉得值得单独写一篇" 是平静过渡，不是口号

## 怎么用这份指纹

跟卡兹克版相同——按目标场域和内容类型选风格：

| 内容类型 | 风格基线 |
|---|---|
| 工具评测、产品发布、技术教程 | 优先抄歸藏（A2 + A3 + B3 + B6）|
| 设计 / 审美 / 方法论自述 | 优先抄歸藏（A5 + B2 + B7）|
| 行业反思 / 立场表达 / 黑暗森林型 | 优先抄卡兹克（A2 + A5 + B5 + B6）|
| 速报 / 开箱 / 第一时间播报 | 看小互（待补，样本不足）|
| 长文综合（既要有情绪又要信息）| 卡兹克 + 歸藏混搭：开场抄卡兹克，主体抄歸藏，收尾再抄卡兹克 |

写作 sub-agent 在 publish-article skill 里被调用时，**应根据稿件主体类型决定调哪份指纹**。

## 反例 / 警惕

- **硬抄"约束越严，风格越稳"型短句会显得装**：他这种价值判断句站得住，是因为后面有"我做了十年设计"这种身份背书。新手没积累就压判断会显得空。
- **段段平铺并列 = 像产品 PRD**：他的并列结构有"它长什么样 / 它能做什么 / 怎么用 / 为什么长成这样" 这种隐性叙事弧度托着，不是真的散点平铺。直接抄列表会变 PRD 体。
- **中英混排是双刃剑**：在他的圈层（设计师 + 开发者 + AI 工具圈）是身份信号，到普通公众号读者那里是劝退信号。**用前先确认目标读者画像**。
- **完全不灌情绪 = 在朋友圈传不开**：歸藏的文章在垂直圈层很受欢迎（GitHub Star 验证过），但很难破圈。如果目标是"出圈"，光抄歸藏不够，至少要在开场和结尾局部抄卡兹克。
