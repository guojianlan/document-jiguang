## A 结构层 7 条指纹

### A1. 内容类型决定结构模板（4 套独立模板）

idoubi 不是"一种风格通吃"。**4 类文章用 4 套结构模板**：

| 类型 | 起手 | 中段 | 收尾 |
|---|---|---|---|
| 产品复盘 | "今年 X 月，我做了 Y" + 一句话定义 + 数字成绩 | 开发历程 → 核心问题 → 平台化展望 | 数据复盘 + 招募/合作 |
| SOP 教程 | "最近 X 很火 / 我想给 X 降降火" | `### 场景名` 平铺，每场景 1 段+1 图 | 选型建议 + 一句金句 |
| 技术评测 | "X 是什么 + 协议规定" 直接定义 | 流程 → 实现代码 → 利弊 → 适用场景 | 引出下一节（"既然 X 这么麻烦，是不是应该有 Y？""当然有"）|
| 年度总结 | "这是我从 X 以后的第 N 篇" | 按方向罗列项目，每项目数据+决策+反思 | "我最想做的方向是 Z" + 性格金句 + 祝福 |

教程篇的"我们应该如何用好 X"、复盘篇的"产品名 + 一句话定义"、年总篇的"过去两年我基本上把 X 都做了一遍"——**这些是模板里的固定零件，不是即兴**。

### A2. "时间 + 我做了 X" 的开场公式

复盘文几乎都是这个模式：

- "今年 3 月，我做了一个 AI 搜索引擎，名字叫做 ThinkAny"
- "最近三个月，我用 Claude Code Vibe Coding 了几个项目"
- "2024 年 11 月底上线了 MCP.so"
- "1 月份的时候，我想做一个开源版本的桌面 Agent"

**时间锚定 + 第一人称行动 + 产品名加链接**——三件套钉死开场。比卡兹克的"昨天，X" 更硬，因为带了产品名和具体月份，是**履历式开场**而不是新闻式开场。

### A3. 一句话定义 + 一句话定位 + 一句话成绩

新产品/新概念出场必带的"三连定义"：

> "ThinkAny 是一款新时代的 AI 搜索引擎，利用 RAG 技术快速检索和聚合网络上的优质内容..."（是什么）
> "ThinkAny 的目标是：搜得更快，答得更准。"（目标）
> "ThinkAny 的定位是做全球化市场..."（定位）

复盘文连续三五段都是 "ThinkAny 的 X 是 Y" 句式。**让读者快速把陌生名字拼装成形象**。

### A4. 数字优先，自夸克制

凡是产品成绩，数字精确到位：

- "月访问量 60 万的全球化产品，用户覆盖日本/埃及/俄罗斯/巴基斯坦等国家和地区，累计用户数突破 17 万"
- "MCP.so 网站总访问量 11M，新增用户 1.5M。广告收入 1.2w 美金"
- "CopyWeb 2025 年 ARR 2.4w 美金，有点意外"
- "10 月份收到了第一笔版税：2.3w。三个月累计销售 3700 余册"
- "github star 涨到了 1.4k"

**数字本身就是表达**。不写"成绩很不错"，写"2.4w 美金，有点意外"。**数字带括号备注、带情绪后缀**（"有点意外""算是亏本""相当炸裂"），是他独有的诚实账本风。

### A5. 主动暴露失败、收益、决策回退

这是 idoubi 跟标准技术博客最大的区别：

- "ROI 没打正，是一笔亏本的投资"（写书复盘）
- "因为合规和用户隐私方面的问题不太好解决，这个项目后面也就搁置了"（ChatSum）
- "可惜因为个人精力分散、时间有限，没有做得更好"（MCP.so）
- "20w 美金收购，我没有卖"（事后看是后悔）
- "用户留存不高，也没有很好的商业化"（产品自检）

**失败/瓶颈/收益数字与产品介绍同等比重**。这是"诚实复盘体"的核心——读者读到的不是营销稿，是私董会笔记。

### A6. 产品列表式叙事（多产品单文）

vibe-coding-projects、my-ai-2025、my-ai-projects-in-2023 都是这个模式：

```
[产品名 + 链接]
[一句话由来：X 火了，我想做 Y / 看到 X 痛点，于是 ...]
[做法 / 架构 / 截图]
[数据 / 反思 / 后续打算]

[下一个产品名 + 链接]
...
```

**每个产品 200-500 字独立卡片**，跟做项目卡片库一样。读者可以扫读跳读，也可以专挑感兴趣的产品深读。**这是独立开发者多线作战的结构反映**。

### A7. 设问式推进（技术解析专属）

mcp-transport 篇 31 个问号——**通过自问自答推进**：

> "既然 stdio 传输访问远程资源这么麻烦，是不是应该有一种更适合远程资源访问的传输机制？"
> "当然有。可以使用 SSE 传输。"

这是技术写作的招牌句型，**用问句把读者拉到下一个章节**，比"让我们看下一节"或"我们继续讨论"自然得多。教程篇也有变体：开篇"我们真的已经进入了'人人皆可做产品'的时代了吗？" 把问题抛在最前。

## B 语言层 7 条指纹

### B1. 英文术语零翻译，原文保留 + 反引号包裹

mcp-transport 篇 180 个英文缩写、36 个反引号术语。**默认读者是同行**：

- `MCP / RAG / API / SaaS / MVP / PMF / UV / PV / ARR / MRR / K8S / LLM`——一律不翻译
- `Claude Agent SDK` / `OpenClaw` / `Plan + Act` / `Function Calling`——产品名/概念名直接英文
- `Tab` / `Ctrl + L` / `Cmd + I`——快捷键原文，不写"Tab 键"

跟卡兹克"小龙虾"指代 Claude 的圈层化是反方向的：**idoubi 不造黑话，他直接用官方英文术语**。这让他的文章在国际化技术圈和英文 SEO 上有红利，但对小白读者门槛偏高。

### B2. 行动动词第一人称密集叙事

"我做了 / 我写了 / 我开源了 / 我发布了 / 我实现了 / 我尝试了 / 我选择了"——vibe-coding 文 12 次。

跟卡兹克的"我看见了 / 我惊了 / 我研究了"是不同的"我"：**idoubi 的"我"是动作主语，卡兹克的"我"是观察者主语**。idoubi 在写"我做了什么事"，卡兹克在写"我看到了什么事"。

### B3. 数字+量级+货币单位，不绕弯

"5k 美金的部署成本"、"3 个 05 后少年，四个月做到了三百万美金的 ARR"、"1500 本，支出 6w 多"、"日均 UV 3k，日均 PV 20k"——**金钱、用户、增长速度全部具象到精确数字**。

不写"成本不低"，写"5k 美金"。不写"用户增长很快"，写"4 月份访问量 2.7M"。**钱和数据是他文章的硬通货**，比观点更有说服力。

### B4. 列举式短句，每条独立成段

```
1.   是我很感兴趣的方向
2.   产品有价值，能带来成就感
3.   在我的能力范围内
```

或：

```
*   不喜欢社交就不社交
*   组不了团队就独立开发
*   拿不到投资就自己赚钱
*   不想专注就广泛尝试
```

**列表是 idoubi 最重要的修辞工具**。无论分析、复盘、建议、决策，**第一反应就是列点**。这让段均字数压低、扫读成本低，但也让文章看起来像项目文档而不是文章。

### B5. 因果链式叙事，"于是 / 所以 / 因此 / 后来" 高频

"OpenClaw 自带的 Web UI 有点丑。... **于是** 我开源了 ChatClaw"
"半个月前，微信开放了 ClawBot... **于是** 我开源了 WeClaw"
"GPTalk 用户流失了。... **后来** 我做了知了阅读"
"我分析了一下... **所以** 我得出的结论是..."

**强因果链**——每个动作都有上游触发器、下游效果。读者跟着他的决策树走，比看观点输出更有代入感。

### B6. 一句话价值判断，不长篇大论

> "MCP 是原子能力的包裹器，外部工具的说明书。"
> "AI 搜索的第一要义是准确度。"
> "营销是关键。"
> "做难而正确的事情，长期主义。"
> "性格决定命运。"

**短句压判断、不展开**。判断句单独成段、加粗或引用块。读者读到这种句子会自动当成 takeaway 截图传播。**比卡兹克的"恐怖如斯"更工整，但也更安全、更少爆点**。

### B7. 收尾"性格自洽 + 一句口号"

不是卡兹克的"长 → 短 → 口号"三段式，而是 idoubi 的"个人状态接受 + 朴素口号"两段式：

- "All in AI，未来可期。"（辞职文）
- "做难而正确的事情，长期主义。/ 虽然前路艰险，依然砥砺前行。"（ThinkAny 复盘）
- "相信自己，你才是那个最棒的筑梦师。"（Cursor 教程）
- "不喜欢社交就不社交，组不了团队就独立开发... / 与其求而不得，感受痛苦。不如顺其自然，享受自由。"（年总）
- "新的一年，少一些宏大叙事，多一些落地执行。"（年总）

**收尾不追求戏剧性高潮，追求"我接受了我自己"的自洽感**。这跟"独立开发者""自由职业"身份呼应。读者读完不是被点燃，是被安抚。

## 一句话定位

**idoubi 是"诚实账本派"——独立开发者复盘体的代表，技术决策与商业数字并重，靠数字密度和因果链推进，靠性格自洽收尾。**

不是"诚实复盘体"那么简单——他**不止复盘，还把每个决策的上游触发、下游产出、ROI 数字、后续打算列成账目**。读者拿到的不是故事，是可抄的项目流水。

## 改写示例

取本仓库 12 components 文章原段：

> 原文：
> GitHub 上 shareAI-lab 的 `learn-claude-code` 把这件事掰得很清楚——12 节课，12 个递进的机制，每章在前一章基础上叠加一个东西。我读完 12 章源码（每章 100-800 行 Python，加起来 4000 多行），读完反而发现，外界普遍流传的那张"12 组件平铺图"是错的。

按 idoubi 风格重写：

> 4 月份的时候，我读了 [shareAI-lab/learn-claude-code](https://github.com/shareAI-lab/learn-claude-code) 这个仓库，1.4k star。
>
> `learn-claude-code` 是一个 Claude Code 内部机制的逐章拆解项目，12 节课，每节 100-800 行 Python，加起来 4000+ 行代码。
>
> 我花了一个周末，把 12 章全部读完。
>
> 读完最大的发现是：网上流传的"Claude Code 12 组件平铺图"是错的。
>
> 主要问题有两点：
>
> 1.   12 个组件不是平铺关系，而是递进关系
> 2.   平铺图忽略了组件之间的状态依赖
>
> 这导致很多基于平铺图设计的 Coding Agent 项目，从架构层面就走偏了。

差异点：
- 加 "4 月份" 时间钩子
- 仓库名加链接、加 star 数

## 反例 / 警惕

- **数字凑数会翻车**：idoubi 数字之所以可信，是因为都是真账。如果没有真实数据，硬编"ARR 2 万美金"会比不写更糟。
- **失败暴露不等于卖惨**：他暴露失败是为复盘，配套的是"后续打算"或"得出的结论"。只暴露不复盘 = 卖惨。

- **英文术语不是越多越好**：他的术语密集是因为读者画像是开发者。给非技术读者写就别这么干。
- **性格金句不能强凑**：他的"不喜欢社交就不社交"是真心话。没活到那个状态硬抄会显得装。
