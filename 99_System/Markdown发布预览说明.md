# Markdown 发布预览说明

这个说明解决的是一个非常实际的问题：

`Markdown 里的本地图片路径，直接复制到飞书文档时，通常不会变成真实图片。`

## 为什么会这样

- Markdown 文件里的图片本质上只是引用路径
- 飞书在粘贴时不会自动读取你本地磁盘里的文件并上传
- 即使把链接从 `.svg` 改成 `.png`，如果它仍然只是本地路径或本地 URL，飞书通常也不会认成图片内容

所以“换成 PNG 路径”不等于“可以直接贴过去”。

## 更稳的方式

更稳的测试路径是：

1. 把 Markdown 渲染成浏览器里的 HTML
2. 尽量把本地图片内嵌到 HTML 里
3. 再从浏览器复制富文本内容到飞书文档

如果 Markdown 原文引用的是 `SVG`，预览服务会优先查找同名 `PNG` 并改用它，这样更符合飞书等平台的图片格式要求。

当前预览脚本支持常见 Markdown 结构，包括：

- 标题
- 段落
- 有序列表和无序列表
- 引用块
- 行内代码
- fenced code block
- 表格
- 分隔线
- 本地图片与链接

## 本地预览服务

仓库提供了一个本地服务脚本：

```bash
node scripts/serve_markdown_publish_preview.js
```

如果想直接打开某一篇文章：

```bash
node scripts/serve_markdown_publish_preview.js "03_Outputs/Drafts/2026-03-23_article_AI Vibecoding到底是什么_发布版.md" --open
```

默认端口：

- `4312`

如果希望把端口固定到仓库配置里，可以在仓库根目录 `.env` 中设置：

```bash
MARKDOWN_PUBLISH_PREVIEW_PORT=4312
```

优先级约定：

- 命令行 `--port` 优先
- `.env` 里的 `MARKDOWN_PUBLISH_PREVIEW_PORT` 其次
- 默认值 `4312` 最后

也可以自定义：

```bash
node scripts/serve_markdown_publish_preview.js "03_Outputs/Drafts/xxx.md" --port 4313 --open
```

## 监听模式

如果你正在开发预览服务本身，希望在修改脚本后自动重启，可以使用 `nodemon`：

```bash
npm run preview:md:watch -- "03_Outputs/Drafts/xxx.md" --open
```

当前监听范围包括：

- `scripts/serve_markdown_publish_preview.js`
- `scripts/restart_markdown_publish_preview.js`
- 仓库根目录 `.env`

这个模式更适合改预览器逻辑本身，不是给普通内容编辑阶段日常使用的。  
因为 Markdown 正文内容本来就是按请求实时读取，改文章通常不需要重启服务。

## 重启服务

仓库提供了一个重启脚本，会按当前端口配置自动停掉旧进程并拉起新服务：

```bash
bash scripts/restart_markdown_publish_preview.sh
```

如果希望重启后直接定位到某篇文章，也可以把文章路径继续传进去：

```bash
bash scripts/restart_markdown_publish_preview.sh "03_Outputs/Drafts/xxx.md" --open
```

脚本会在仓库 `.tmp/` 下写入日志和 pid 文件，便于后续排查。

## 推荐用法

1. 启动预览服务
2. 在浏览器里选择要发布的 `.md`
3. 先在侧边栏里选择目标平台
4. 页面正文仍然按正常阅读样式展示
5. 如果要定向复制，先框选正文内容，再点击“复制当前选中内容”
6. 直接粘贴到目标平台测试

如果你正在修改预览器脚本本身，可以把第 1 步换成 `npm run preview:md:watch`。

## 飞书复制版说明

预览页现在区分两类输出：

- 页面正文：主要服务于浏览器内阅读和结构检查
- 平台复制版：主要服务于剪贴板复制和目标平台粘贴测试

这样拆开的原因是：

- 网页展示可以依赖样式表
- 飞书粘贴更看重 HTML 结构和行内样式
- 同一份 DOM 同时兼顾“网页好看”和“飞书稳定”时，代码块最容易出问题

当前预览页已经支持：

- 选择目标平台，例如飞书、通用富文本
- 页面正文保持正常展示，不因平台切换而改变阅读样式
- 点击按钮时，优先复制当前选中的正文内容；如果没有选中内容，就复制当前正文全文
- 正常的 `Command+C` 仍然保持浏览器默认行为，不再由预览页拦截

当前飞书复制版的处理策略是：

- 保留标题、段落、列表、表格、图片等常见结构
- 保留行内代码样式
- 把块级代码从网页预览结构中拆出来，改成带行内样式的内容块

这样做的目标不是保留飞书原生代码块语义，而是优先减少下面这些常见问题：

- 代码块里又叠一层行内代码背景
- 页面看起来正常，复制进飞书后样式完全变形
- 复制后代码块直接丢失或被拆碎

## 当前边界

这个方案已经比“直接复制 Markdown 源码”更接近可用，但仍有边界：

- 飞书是否接受浏览器复制过来的内嵌图片，取决于它自己的粘贴策略
- 如果飞书仍然拒绝图片，下一步就不能只靠复制，而要改成“上传图片再插入正文”的流程

## 适用场景

- 飞书文档发布前预览
- 公众号正文拷贝前检查
- 内部知识库内容转富文本
