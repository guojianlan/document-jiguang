# critique

这个 skill 解决的问题是：AI 写出来的中文常常"读起来通顺、读完一无所获"。它不替你重写整篇，而是**抓住低分维度强制返工**，让产物从"看起来像文章"升级到"读者愿意读完并转发"。

## 何时使用

- 用户产出文章草稿、发布版、主题笔记、社交切图文案后
- `/publish-article` 流程末尾，正文落盘前
- 用户说"读起来怪怪的"、"AI 味太重"、"没干货"、"不像我写的"
- 复盘历史发布物料、判断哪些值得保留、哪些要重写

不适合：还没成文的素材整理、单条短句、纯数据表格。

## 输入

- 一篇 Markdown 文章路径（必填）
- 可选：**目标场域**（决定规则集严格度，详见 §1.5）。如未声明，从文件路径或用户提示推断；推断不出走 `blog` 模式
- 可选：用户期望的目标读者画像（默认从正文推断）
- 可选：参考样本——用户自己写得好的旧文（用来做"本人语感对照"）

## 资源加载规则

执行前必须先读以下两个参考表，扫描时按表逐项核对：

- [references/zh-ai-tells.md](./references/zh-ai-tells.md) — 中文 AI 痕迹高频词与硬规则（14 节 + 8 项快速自检）
- [references/rule-taxonomy.md](./references/rule-taxonomy.md) — 37 模式分类（P1-P37）与维度映射

## 工作流

### 1. 全文先读，再下手

不要边读边评。先一次完整读完，记下两个直觉：
- 读完一句话能复述出来的核心是什么
- 哪段读到一半想跳过

直觉记录会成为后续打分的锚点，避免逐段微观打分时丢掉整篇视角。

### 1.5 判断目标场域 → 决定规则集

不同场域读者宽容度不同，规则严格度必须分层。**这是 critique 的核心适配机制**——以前一刀切导致用户本人的"而是"句、长段方法论被当 AI 味删掉。

#### 三档场域

| 场域档 | 触发条件 | 适用规则集 |
|---|---|---|
| **`strict`**（严格）| 公众号 / 小红书 / 朋友圈 / 抖音文案 / 微博长文 | 全部 14 节硬规则 + 全部快速自检 10 条 |
| **`blog`**（默认）| 个人博客 / 知乎专栏 / 掘金 / 少数派 / 内部 wiki / 技术文档 | 跳过 §B-skip 标注的规则；其他规则照执行 |
| **`reference`**（参考类）| AGENTS.md / SKILL.md / 99_System / wiki/topics 等 schema/ref 文档 | 只跑 §universal 标注的规则；不跑节奏类规则 |

#### 怎么判断场域

按优先级：
1. 用户在调用时显式声明（"按 strict 跑 / 按 blog 跑"）
2. 文件路径推断：
   - `outputs/drafts/` + 标题含"公众号/朋友圈/小红书" → `strict`
   - `outputs/drafts/` + 默认 → `blog`（除非另有提示）
   - `wiki/` / `99_System/` / `04_Templates/` / `AGENTS.md` 等 → `reference`
3. 用户在 AGENTS.md / 风格库 `_self.md` 已声明的"目标平台" → 按其覆盖

**最关键的事实**：用户本人在 `99_System/voice-references/_self.md` 里被验证有稳定的"而是 X，而是 Y"句式（OpenSpec 文 8+ 次），这是他**风格特征不是 AI 味**。所以 `blog` 模式必须放过这条。

#### `blog` 模式跳过哪些规则（§B-skip）

下列硬规则在 `blog` 模式**不计入扣分**，只做软警示（统计次数但不扣阅读节奏分）：

- 第一节"而是"句 ≤ 1（用户标志特征）
- 第九节极值判断 ≤ 1（用户方法论文标题里有"最X的做法"是常见结构）
- 第四节冒号 ≤ 2（方法论 / 教程文需要冒号引出步骤、定义等，硬卡 2 个不现实）

`reference` 模式额外跳过：
- 整个第二、三、五、六、十三、十四节（全部"路标词 / 伪口语 / 二人称连续 / 步骤承接 / 过渡腔 / 空范围词"——这些规则是发布稿读者视角的，schema/ref 文档不需要）

#### `universal` 规则（所有场域都跑）

下列规则**所有场域**都执行，不论模式：

- 第三节客服 / AI 身份口吻（"作为 AI / 希望这能帮助你"等）零容忍
- 第八节戏剧化揭露修辞（"扯下遮羞布 / 撕下面具"等）零容忍
- 第十节伪学术腔（"舆论场 / 宏大叙事"等）零容忍
- 模板段（挑战 / 机遇 / 展望三段式）零容忍

### 2. 第一轮扫描：跑硬规则与 37 模式

对照 `zh-ai-tells.md`：

- 数"而是"出现次数
- 数冒号数量
- 数路标词数量
- 扫 AI 高频隐喻词、伪口语化词、降维引导语、戏剧化揭露、极值判断
- 检查二人称连续、协作路标、客服身份口吻、模板段、"一段一标题"

对照 `rule-taxonomy.md`：

- 用 P1-P37 逐项过一遍，命中即记
- 把命中按"评分映射"表归到 4 个维度

### 3. 按 4 个维度打分（每项 1-5 分）

#### A. 信息密度
- **5 分**：每段都至少给出一个读者之前没法直接 google 出来的判断、数字、对照、亲历细节
- **3 分**：观点对，但读者从标题就能猜到结论
- **1 分**：通篇是"重要性 + 必要性 + 应当 + 因此"的注水句

证据：直接引用正文中信息密度最低的句子，并标注命中的 P 编号（如 P1 / P5 / P22）。

#### B. 独特视角
- **5 分**：有一个明显是"作者本人才会这样想"的判断 / 比喻 / 逆向角度
- **3 分**：观点正确但通用、换任何一篇同类文章都成立
- **1 分**：百科式综述、ChatGPT 默认输出口径

证据：找出最像"AI 默认答案"的那段，引用 + 标注 P 编号（P6 / P29 / P32 等）。

#### C. 可操作性
- **5 分**：读者读完能立刻做 1-3 件具体的事，且事项细到工具 / 路径 / 参数级
- **3 分**：给出方向但停在"建议关注 X"
- **1 分**：只有抽象框架、没有落地动作

证据：找出抽象度最高、最不可执行的句子。

#### D. 阅读节奏
- **5 分**：长短句交替、有停顿感、有"啰嗦但有人味"的细节、有节奏断点
- **3 分**：句长均匀、过渡词标准（首先 / 其次 / 最后 / 总之）、每段开头都用判断句
- **1 分**：通篇排比 + 长句 + "不仅...还..."、读起来像念稿

证据：连续引用 3 句以上呈现节奏问题的段落 + 标注命中的 P 编号（P10 / P13 / P30 / P9 等）。

#### 自动扣分规则（按场域分支）

**所有场域都生效**：
- `zh-ai-tells.md` 快速自检 universal 项命中 ≥ 3 → **阅读节奏自动 ≤ 2 分**
- 协作路标词 ≥ 1 次 → **独特视角减 1 分**
- 模板段（挑战 / 机遇 / 展望）出现 → **独特视角自动 ≤ 2 分**
- 客服 / AI 身份口吻 ≥ 1 次 → **整体扣 1 分**
- 戏剧化揭露修辞 ≥ 1 次 → **独特视角减 1 分**

**仅 `strict` 模式额外扣分**：
- "而是" ≥ 2 次 → 阅读节奏减 1 分
- 极值判断 > 1 次 → 阅读节奏每超 1 次减 0.5
- 冒号 > 2 次 → 阅读节奏减 1 分
- 二人称连续使用 → 阅读节奏减 1 分（含对仗变体）

**`blog` 模式**：上述 strict 项**只统计次数列在诊断卡里做软警示**，不实际扣分。但如果同一规则**远超用户基线均值的 3 倍**（比如方法论文里"而是"用了 25 次而 `_self.md` 基线是 8 次），仍触发警示。

**`reference` 模式**：跳过节奏类全部扣分，只保留 universal 5 条。

### 4. 出诊断卡

按下面格式贴回给用户（不要藏在文件里，先口头摆出来），**第一行必须显示场域档**：

```
场域：strict | blog | reference

信息密度  X/5  → 证据："..."（命中 P? P?）
独特视角  X/5  → 证据："..."（命中 P? P?）
可操作性  X/5  → 证据："..."
阅读节奏  X/5  → 证据："..."（命中 P? P?）

硬规则命中：而是×N | 冒号×N | 路标词×N | 协作路标×N | 隐喻词×N
[blog 模式时额外行] 软警示（不扣分）：而是×N | 极值×N | 冒号×N
```

不要给安慰分。某项是 2 分就写 2 分。

### 5. 决定返工范围

- 任一维度 ≤ 2 → **必须返工该维度对应的段落**
- 全部 ≥ 4 → 通过，不强制返工，但可建议精修
- 总分 < 14（4 项均分 < 3.5） → 标记为"应整篇重写而非局部修补"

返工不是再润色一遍。每个低分维度有专属补救动作：

| 低分维度 | 补救动作 |
|---|---|
| 信息密度低 | 找出最虚的 3 句，要求用户补具体数字 / 案例 / 亲历，或直接删掉 |
| 独特视角弱 | 找出 1 个"作者会怎么想得不一样"的反向角度，重写开篇判断 |
| 可操作性差 | 把每个抽象建议落到工具名 + 具体步骤 + 失败兜底 |
| 阅读节奏差 | 拆长句、删过渡词、塞 1-2 个生活化细节、加 1 个断点 |

### 6. 执行返工

按补救动作改 Markdown 文件。改的时候：
- **不要全文重写**，只改打中证据的段落
- **保留原意**，不要为了节奏改掉作者的核心判断
- **改完贴 diff**：让用户看清改了什么、为什么改

返工时直接执行 `zh-ai-tells.md` 第 1-12 节的改写策略：
- "而是"句去前半句否定 → 留判断
- 协作路标词、客服口吻直接删
- 戏剧化揭露 → 直接给事实
- 步骤承接过渡句 → 删前半句，新事实起笔
- 极值判断 → 紧跟可核对事实

### 7. 二轮自审（"obviously AI" 复扫）

返工后做一次**第二遍审视**（结构借鉴自 blader/humanizer）：

1. 把返工后的版本贴出来
2. 自问："这段读起来还有哪些地方明显是 AI 写的？"
3. 简短列出剩余的痕迹（节奏、模糊、套话）
4. 再改一轮，目标是"不再明显是 AI"
5. 输出最终版

二轮只看返工过的维度，不重打全部 4 项。如果二轮仍 ≤ 2，**停手**，把问题贴给用户人工决策，不要无限自动重写。

### 8. 回写规则

如果在打分或返工中发现了新的"AI 味模式"（典型词、典型结构、典型注水句），同步更新：

- [zh-ai-tells.md](./references/zh-ai-tells.md)（新增条目，标注命中场景）
- [rule-taxonomy.md](./references/rule-taxonomy.md)（如果是新模式，扩展到 P38+）
- [04_Templates/T-反AI味检查清单.md](/Users/apple/Desktop/project/document/04_Templates/T-反AI味检查清单.md)
- 必要时更新 `AGENTS.md` 的稳定偏好节
- 必要时更新 `99_System/Skill候选池.md` 中本 skill 的成熟度

按 `AGENTS.md` 规则 A：只改代码不回写说明视为未完成。

## 边界

- 这是**校验 + 局部返工** skill，不替你写新文章
- 不做事实核查（事实问题应在 `/publish-article` 之前的来源整理阶段就锁死）
- 不做 SEO、关键词密度、社交平台合规这些发布层关注点（那些归 `/publish-article`）
- 评分主观，不要把它当成绝对标准——它的价值是逼你**用读者视角再读一遍**，并按 37 模式 + 中文硬规则做客观对照

## 与其他 skill 的协作

- 与 `/publish-article` 串联使用：发布包装产出后立即跑 `/critique`
- 与 `/render-svg` 解耦：本 skill 只看文字，视觉资产问题归 `/render-svg`
- 与 `/verify` 解耦：本 skill 只看内容质量，代码与 SVG 健康度归 `/verify`
