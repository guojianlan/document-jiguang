# 导出链路一致性（方法论）：用"中间层模型 + 字体度量 + 双通道渲染"把 PPTX 与 PDF 高保真对齐

## 问题概述

将网页中的幻灯片导出为 PPTX 或 PDF 看似简单，但实际面临多重挑战：

- 同一页面导出格式呈现不一致：文本换行、行距、对齐、图片裁剪等存在差异
- 多语言场景失效：RTL 文本（阿拉伯语/希伯来语）出现方向、排序混乱
- 字体控制问题：缺字、乱码、Emoji 显示为空框
- 维护困难：修复某个格式会影响另一个格式

本文提出一套**可长期维护的导出方法论**，将"输出一致性"转化为明确的工程目标。

---

## 核心方案（TL;DR）

1. **中间层模型（IR）**：从 DOM/样式抽取统一的"Presentation/Slide/Element"结构，两种导出共享同一输入
2. **单位统一**：采用物理单位（英寸）作为主单位；PDF 使用点数（pt），PPTX 天然使用英寸
3. **文本度量**："不靠猜，靠测量"——用 fontkit 测量字形宽度；实现字体回退；PDF 字体必须子集化
4. **BiDi/RTL 处理**：在行级别处理双向文本重排与镜像字符，确保正确性与可控性
5. **回归体系**：通过像素 diff、pHash、性能指标等观测手段，使一致性成为可维护的工程能力

---

## 单位体系设计

```
设计宽度合同：
- DESIGN_WIDTH_PX = 1161px
- DESIGN_WIDTH_IN = 10 inches
- 推导：pxToIn(px) = px * (10 / 1161)
- PDF 转换：1 inch = 72 points
```

**工程建议**：允许"精确值"与"历史值"并存。使用精确计算支撑页面尺寸与坐标系，用历史值兼容视觉敏感项（字体大小、行距），通过开关对比决定何时收敛。

## 关键技术实现

### 1. DOM 到中间层转换

- 遍历每个 slide 元素，抽取文本、图像、形状、SVG
- 统一位置/尺寸：均转换为英寸单位
- 保留页面 px 尺寸用于后续映射

### 2. 文本一致性的字符级处理

**断行与度量流程**：

```
预处理（复杂脚本 shaping）
  ↓
字符级度量（fontkit + fallback 字体）
  ↓
Token 化断行（按宽度、词边界、CJK）
  ↓
行级 BiDi 重排（embedding levels + 镜像字符）
  ↓
逐字符渲染
```

### 3. 图片处理策略

- **Cover/Contain**：用统一公式计算缩放与偏移，避免两个格式各自实现
- **透明度处理**：用 Canvas 合成 PNG，将透明度效果"烘焙"进位图
- **体积优化**：PDF 可选"保留原图+裁剪路径"或"直接输出裁剪后小图"

### 4. SVG 与形状处理

决策策略：
- 简单形状（矩形、直线）：优先矢量化
- 复杂 SVG（mask、filter、混合模式）：Rasterize 为 PNG，使用更高像素倍率
- 明确降级边界：单元素失败可跳过，不中断整体导出

### 5. RTL/BiDi 系统化治理

**三层架构**：
1. 方向开关：根据 locale 设置 `rtlMode` 参数
2. 对齐翻转：`start/end` 映射到 `left/right`（RTL 下翻转）
3. 行级重排：对每一行做 BiDi 重排，镜像字符替换（括号等）

**关键原则**："不能整段一次性重排"——必须先按逻辑顺序断行，再逐行重排，否则会产生乱码。

## 指标与验证体系

### 三类核心指标

**正确性指标**：
- 像素 diff 超阈值的页数比例
- 缺字率（notdef glyph 数 / 总数）

**性能指标**：
- 导出耗时 p95（按页数分桶）
- 内存峰值
- 失败重试次数

**产物指标**：
- 文件体积 p95
- 字体子集化占比
- 位图元素占比（影响可编辑性）

### 通过标准建议

- 样本库像素 diff 不回退；RTL/混排样本不跑版、不乱序
- notdef 缺字率接近 0，fallback 命中可追踪
- 导出耗时与体积 p95 不回退

### 最小可复现验证

1. 准备 10–50 份代表性文档（多语言、复杂 SVG、长段落、图片裁剪）
2. 记录导出耗时/体积/错误
3. PDF 逐页渲染为位图，与浏览器截图做像素 diff（允许 1–2px 容差）
4. 每次改动都跑回归套件

## 常见问题解答

**Q：为什么不直接截图导出？**
A：失去可编辑性与文本语义（不可选、不可搜、不可复制）。

**Q：如何避免缺字？**
A：按 Unicode 范围准备 fallback 字体（CJK、Arabic、Hebrew、Emoji 等）；字符级测量时自动切换。

**Q：字体子集化的必要性？**
A："完整字体可能 MB 级，多字体全量 embed 会爆炸文件体积。子集化只 embed 实际使用的字符集合。"

**Q：BiDi 为什么必须行级处理？**
A："换行是布局的一部分。先逻辑顺序断行，再逐行视觉重排，否则断行点会错误导致乱码。"

**Q：需要 100% 一致吗？**
A："不同阅读器本身有渲染差异。目标应是：关键版式一致（不多行/不跑版）、误差可量化、改动不回退。"

**Q：像素 diff 阈值如何定？**
A："允许 1–2px 容差（抗锯齿差异），但'多一行/少一行/对齐反了/顺序乱'设为硬失败。"

**Q：样本库需要多少份？**
A："至少 10–50 份，覆盖长段落、列表、混排、emoji、图片裁剪、复杂 SVG。每次线上翻车都追加样本。"

## 边界与降级策略

- **复杂 SVG**：优先 rasterize；失败时可跳过单元素
- **未支持 shape**：用最接近的基本形状降级，或转换为图片
- **Emoji**：fallback 到 emoji 字体；不支持时退化为黑白或位图
- **超大文档**：按页流式渲染、进度回调、必要时限制页数

---

## 适用场景与前置知识

**适合对象**：需要导出 PPTX/PDF 的 Web 编辑系统，对排版一致性有要求

**不适合**：只需"能导出"不在乎细节，或接受截图式导出

**前置知识**：DOM/CSS 基础、px/pt/in 概念理解；无需深入掌握 PDF/PPTX 结构

---

**核心思想**："把一致性变成可观测、可回滚的工程能力，而非永远修不完的黑盒。"
